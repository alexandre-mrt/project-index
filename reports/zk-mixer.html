ZK Payment Mixer

Privacy-preserving ETH mixer using zero-knowledge proofs

Circom Solidity Groth16 Poseidon TypeScript React snarkjs

The Challenge

Every ETH transfer is publicly traceable.

Sender address, receiver address, and amount are all visible on-chain. Anyone with a block explorer can trace the full flow of funds between wallets.

When Alice sends 1 ETH to Bob, that link is permanently recorded on Ethereum. Chain analysis firms, curious observers, and adversaries can all reconstruct the payment graph. There is no native privacy layer for value transfers on public blockchains.

The mixer pattern solves this by inserting a cryptographic "break" between deposit and withdrawal. Many users deposit the same fixed amount into a shared pool. Any depositor can later withdraw to a fresh address, and nobody can link the two.

Why It Matters

  • Transaction privacy is a fundamental property of any sound financial system. Without it, every wallet balance and payment is surveillance-ready.
  • Tornado Cash proved the concept at scale on Ethereum mainnet. This project is a ground-up reimplementation to understand the cryptographic and smart contract mechanics.
  • Fixed denomination (0.1 ETH) ensures all deposits look identical on-chain, maximizing the anonymity set. The more users in the pool, the stronger the privacy.

How It Works

Deposit Flow

User generates random: secret (256-bit random) nullifier (256-bit random) Computes off-chain: commitment = Poseidon(secret, nullifier) Calls contract: Mixer.deposit(commitment) + 0.1 ETH | v +--------------------------+ | Incremental Merkle | | Tree (depth 20) | | | | leaf[i] = commitment | | | | root updated, stored | | in ring buffer (30) | +--------------------------+ User saves note: zk-mixer-<secret_hex>-<nullifier_hex> (losing it = losing funds)

Withdraw Flow

User provides: note + recipient address (fresh wallet) Client rebuilds Merkle tree from on-chain events, finds the leaf, computes Merkle proof: pathElements[0..19] (sibling hashes) pathIndices[0..19] (left/right flags) Generates Groth16 proof proving: +----------------------------------------------+ | CIRCUIT: withdraw.circom | | | | Public: root, nullifierHash, | | recipient, relayer, fee | | | | Private: secret, nullifier, | | pathElements, pathIndices | | | | Checks: | | 1. commitment = Poseidon(secret, nullifier) | | 2. nullifierHash = Poseidon(nullifier) | | 3. commitment IS a leaf in the tree | | 4. recipient/relayer/fee bound to proof | +----------------------------------------------+ Submits to contract: Mixer.withdraw(proof, root, nullifierHash, recipient, relayer, fee) | v Verifier.verifyProof() --- valid? ---+ | | | no | yes v v REVERT Mark nullifier spent Send 0.1 ETH to recipient Send fee to relayer (if any)

End-to-End Privacy

The ZK proof reveals nothing about which deposit is being withdrawn. The on-chain verifier only confirms that some valid commitment exists in the tree, without learning which one. The nullifier hash prevents double-spending without revealing the original nullifier.

Contract Design

Mixer.sol

Core contract inheriting MerkleTree, ReentrancyGuard, Pausable, and Ownable. Handles deposits (commitment insertion), withdrawals (proof verification + ETH transfer), and pool statistics.

  • Fixed denomination: 0.1 ETH per deposit/withdrawal
  • Duplicate prevention: commitment uniqueness enforced via mapping
  • Double-spend protection: nullifier hash marked as spent before ETH transfer
  • Anti-front-running: recipient, relayer, and fee are bound into the ZK proof
  • Timelock governance: parameter changes require a 1-day delay
  • Optional NFT receipts: soulbound DepositReceipt ERC721 minted on deposit
  • Rate limiting: configurable per-address deposit cap and cooldown

MerkleTree.sol

Incremental Poseidon Merkle tree with depth 20 (1M+ leaf capacity). Uses a ring buffer of 30 roots for in-flight withdrawal tolerance. External Poseidon hasher deployed from circomlibjs.

MixerLens.sol

Read-only aggregator that batches multiple view calls into a single RPC round-trip for dashboard data.

Key Insights

Merkle Tree Membership Proofs

The core privacy primitive: prove that a commitment exists in the tree without revealing which leaf it is. The prover provides the sibling path (20 hashes) and direction bits. The circuit hashes up from the leaf and checks the computed root matches the public root.

The Nullifier Pattern

Each deposit produces a unique nullifierHash = Poseidon(nullifier). On withdrawal, the contract records this hash. If someone tries to withdraw the same deposit again, the hash is already marked as spent. The nullifier itself stays private (only the hash is revealed), so it cannot be linked back to the original commitment.

Anti-Front-Running

The circuit binds the recipient address, relayer address, and fee into the proof via square constraints. Changing any of these values after proof generation invalidates the proof, preventing transaction relay attacks.

Root Ring Buffer

The contract keeps the last 30 Merkle roots. This means a user can generate a proof, and even if other deposits change the root before submission, the proof remains valid as long as it references a root within the buffer window.

Poseidon over Keccak

Poseidon is algebraic (native to the BN254 field), making it orders of magnitude cheaper inside a ZK circuit compared to Keccak. This is why all commitments and Merkle nodes use Poseidon instead of Ethereum's native hash.
