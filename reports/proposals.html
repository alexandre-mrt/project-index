The Challenge

Sui has no standardized way to handle micropayments for web resources.

Coinbase's x402 protocol brings HTTP 402 "Payment Required" to life, but the existing Sui spec is limited to pre-signed transfers with fixed amounts. No metered billing, no on-chain visibility, no cancellation.

Three structural gaps in the current approach:

  • No metered billing — amount is fixed at signing time. Useless for per-token LLM calls or bandwidth billing.
  • No on-chain state — the facilitator holds an opaque signed blob. Payers can't verify what's pending.
  • No cancellation — once signed, payers have no recourse if the facilitator disappears.

Why It Matters

Governance proposals are how blockchains evolve. A SIP (Sui Improvement Proposal) goes through community review, formal specification, and reference implementation before becoming a standard. This particular SIP defines Sui's payment primitive for the HTTP 402 economy.

x402 is the bridge between HTTP and on-chain payments. Every API call, every gated resource, every AI agent interaction could use it. The payment scheme this SIP defines becomes the foundation for how Sui participates in that economy.

The proposal introduces two payment schemes backed by a Move module:

  • exact_sui — fixed amount, settled in full. Simple API paywalls.
  • upto_sui — payer authorizes a ceiling, facilitator settles the actual amount. Metered billing for LLM tokens, bandwidth, compute.

How It Works

x402 Protocol Flow (as proposed in SIP-x402)

    |------------------------->                         |                    |
    |                         |                         |                    |
    |   402 + PAYMENT-REQUIRED header                   |                    |

    |<-----------------------------------------------------------------------|
    |                         |                         |                    |

    |------------------------->                         |                    |
    |                         |   POST /verify          |                    |
    |                         |------------------------>|                    |

    |                         |                         |------------------->|

    |   200 + resource        |                         |                    |
    |<-------------------------|                         |                    |
    |                         |   POST /settle          |                    |
    |                         |------------------------>|                    |
    |                         |                         |  Sponsored PTB:    |
    |                         |                         |  settle_exact() or |
    |                         |                         |  settle_upto()     |
    |                         |                         |------------------->|
    |                         |                         |  Tx digest         |
    |                         |                         |<-------------------|
    |                         |   {success, digest}     |                    |
    |                         |<------------------------|                    |

Key design decisions in this flow:

  • Shared objects — PaymentAuthorization is a shared object, visible and queryable on-chain from creation. Costs ~2-3s consensus latency but enables cancellation and transparency.
  • Dry-run verification — resource is served after dry-run check, not after on-chain settlement. Users don't wait for consensus.
  • Sponsored transactions — facilitator pays gas (~$0.001), so USDC-only payers and zkLogin agents don't need SUI.
  • NonceRegistry — global Table<vector<u8>, bool> for O(1) replay protection across all payers.

SIP Format

A Sui Improvement Proposal follows a structured governance format designed for clarity and completeness:

SectionPurpose
AbstractOne-paragraph summary of what the SIP proposes
MotivationWhy this change is needed, what problems it solves
SpecificationTechnical details: module code, data structures, protocol flows, deployment
RationaleDesign decisions explained: why this approach over alternatives
Backwards CompatibilityImpact on existing systems (none for this SIP)
Security ConsiderationsThreat analysis: replay, trust model, front-running, state growth
Test CasesMinimum test vectors for the reference implementation
Reference ImplementationMove module + TypeScript SDK + facilitator adapter + example server

SIP metadata fields: SIP-Number, Title, Description, Author, Type (Standard), Category (Application), Status, Created date, and References to dependent SIPs.

Key Insights

Why Sui is ideal for x402

Object model — Payment authorizations map naturally to Sui's owned/shared object paradigm. Each payment is a first-class on-chain entity, not a signature blob.

PTBs (Programmable Transaction Blocks) — The entire authorize-split-transfer-emit flow executes atomically in one transaction. No partial states, no intermediate failures.

Sponsored transactions — Facilitator covers gas (~$0.001 per settlement). Payers holding only USDC never need to acquire SUI. Critical for agent adoption.

zkLogin — AI agents create ephemeral wallets via OAuth (Google/Apple) without managing private keys. This is the primary use case x402 targets: autonomous agent payments.

Sub-second finality — Owned object path: ~400ms. Shared object consensus: ~2-3s. Both are fast enough that resource delivery happens on dry-run, not on settlement.

Design tradeoffs acknowledged

  • Shared objects = consensus latency — ~2-3s instead of ~400ms. Acceptable because users get the resource after dry-run, not after settlement.
  • Permanent storage — settled authorizations persist on-chain (~200 bytes each). Shared objects can't be deleted in current Sui. 1M payments = ~200MB.
  • Facilitator trust — for upto_sui, the facilitator can settle at any amount up to max_amount. Main trust assumption. Mitigated by event monitoring and cancellation.
  • NonceRegistry growth — unbounded. Pruning by valid_before is safe but left to reference implementation.

Security model highlights

  • Replay protection — nonce burned on both settlement and cancellation via shared NonceRegistry
  • Access control — only designated facilitator can settle, only payer can cancel (enforced in Move)
  • Front-running resistance — shared object contents are public, but only auth.facilitator can call settle
  • zkLogin blast radius — ephemeral keys have short lifetimes, authorization window capped by valid_before
  • Gas sponsorship abuse — mitigated by rate limiting per payer address and minimum payment thresholds

Future compatibility

SIP-58 (Address Balances) — when implemented, eliminates coin selection. Payers authorize from address balance directly. This SIP is designed to work with both models.
