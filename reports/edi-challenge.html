EDI Challenge

Serverless NFT badge minter on Polygon with dynamic SVG generation and IPFS storage

TypeScript Next.js 16 Solidity Polygon ERC-721 OpenZeppelin 5 Hardhat Pinata/IPFS Ethers.js v6 ELCA Branding

The Challenge

Build a public web app that mints personalized NFT badges to any wallet, with zero wallet connection required.

The ELCA EDI Challenge required creating a system where non-crypto users can fill a form and receive an ERC-721 NFT badge on Polygon. The user provides their name, project, dates, and a photo. The system generates a branded SVG badge, uploads it to IPFS, and mints the NFT to any wallet address. No MetaMask, no gas fees for the user, no blockchain knowledge needed.

This is a gasless, server-side minting architecture. The Next.js API route holds the contract owner's private key and pays gas on behalf of the user. The badge is a dynamically generated SVG with ELCA's branding (orange-red accent, Montserrat font, profile photo), uploaded to Pinata IPFS alongside ERC-721 metadata. The contract is deployed on both Polygon Amoy testnet and mainnet, with a UI toggle to switch networks.

Why It Matters

  • Zero-friction minting: users fill a form and receive an NFT. No wallet setup, no gas payment, no transaction signing. The server handles everything.
  • Dynamic SVG badges: each badge is a unique SVG generated from form data, with gradient backgrounds, profile photos, and ELCA branding. Uploaded as the NFT image to IPFS.
  • IPFS permanence: both the SVG badge image and ERC-721 JSON metadata are pinned on Pinata, ensuring the NFT's content survives independently of any server.
  • Dual network support: testnet (Amoy) for demo/testing, mainnet (Polygon) for production. Users toggle between them in the UI.
  • Security hardened: input sanitization, rate limiting (IP-based), wallet address validation via ethers.js, and XSS-safe SVG generation.

Smart Contract

EDIBadge.sol: minimal ERC-721 with URI storage. The mintBadge(address to, string uri) function is restricted to onlyOwner, ensuring only the server can mint. Each badge gets a unique token ID (auto-incrementing _nextTokenId). A BadgeMinted event is emitted with the token ID, recipient, and metadata URI for off-chain indexing. The totalSupply() view function exposes the current count.

contract EDIBadge is ERC721, ERC721URIStorage, Ownable {
    uint256 private _nextTokenId;

    event BadgeMinted(uint256 indexed tokenId, address indexed recipient, string tokenURI);

    constructor() ERC721("EDI Badge", "EDIB") Ownable(msg.sender) {}

    function mintBadge(address to, string memory uri) public onlyOwner returns (uint256) {
        uint256 tokenId = _nextTokenId++;
        _safeMint(to, tokenId);
        _setTokenURI(tokenId, uri);
        emit BadgeMinted(tokenId, to, uri);
        return tokenId;
    }

    function totalSupply() public view returns (uint256) { return _nextTokenId; }
}

Key Insights

Server-Side Minting Eliminates UX Friction

The user never interacts with the blockchain. The Next.js API route holds the contract owner's private key (via environment variable) and calls mintBadge() directly. The user only fills a form. This design trades decentralization for accessibility: non-crypto users can receive NFTs without understanding gas, wallets, or transaction signing.

7-Step Minting Pipeline

The /api/mint route executes 7 sequential steps: (1) rate limit check, (2) input sanitization, (3) wallet address validation via ethers.isAddress(), (4) photo upload to Pinata if provided as data URL, (5) SVG badge generation, (6) badge + metadata upload to IPFS, (7) on-chain mint. Each step fails fast with an explicit error message. The response includes the token ID, transaction hash, and explorer URL.

Defense in Depth

Multiple security layers protect the minting endpoint: IP-based rate limiting prevents spam, input sanitization strips dangerous characters and enforces length limits, XML escaping in SVG generation prevents XSS injection, and ethers.isAddress() validates wallet addresses before any on-chain call. The contract's onlyOwner modifier ensures only the server can mint.

Live Badge Preview

The BadgePreview component renders the exact same SVG that will be minted, updating in real-time as the user types. This WYSIWYG approach builds confidence: what you see in the preview is exactly what gets pinned to IPFS and linked to the NFT.
