sui-rps-seal

Provably fair Rock Paper Scissors on Sui — comparing Seal encryption vs Nautilus TEE for hidden moves

Sui Move TypeScript Rust Seal IBE BLS12-381 Nautilus TEE Ed25519 Next.js

The Challenge

On-chain games are transparent. Every transaction is public. In a naive RPS contract, your opponent can read your move from the transaction data before they commit theirs — making the game trivially exploitable.

The fundamental problem: blockchains provide integrity and availability, but not confidentiality. Any game requiring simultaneous secret commitment (poker, RPS, auctions) needs a mechanism to hide data until a reveal condition is met.

This project implements two distinct solutions to the same problem, deployed side by side on Sui testnet, enabling a direct architectural comparison.

Why It Matters

Two fundamentally different cryptographic approaches to on-chain privacy, each with distinct trust assumptions:

V1 — Seal Decryption

  • Moves encrypted client-side with Seal IBE (identity-based encryption)
  • On-chain seal_approve gates decryption — keys released only after both players commit
  • Resolver decrypts off-chain, submits plaintext + salts
  • Contract verifies sha3_256(move || salt) matches stored hash
  • Trust: Seal key servers (threshold BLS12-381)

V2 — Nautilus TEE

  • Same Seal encryption for move commitment
  • Decryption happens inside an AWS Nitro Enclave
  • Enclave computes winner, signs result with Ed25519
  • Contract verifies signature + freshness (5 min window)
  • Trust: TEE attestation + registered enclave key

Game Lifecycle (shared)

  1. Create — Player1 deposits stake (SUI), sets timeout (1 min to 24 hrs)
  2. Join — Player2 matches stake exactly (self-join blocked)
  3. Commit — Both players submit Seal-encrypted moves; game advances to COMMITTED
  4. Resolve — Decrypted moves verified, winner computed, stakes settled
  5. Cancel — After timeout: full refund if no opponent, 50/50 split if unresolved

Security Patterns

Defense in depth (V1): Moves are Seal-encrypted AND sha3-hashed with a random 32-byte salt. Even if Seal were compromised, the resolver cannot submit fake moves — the hash binds move+salt cryptographically.

Defense in depth (V2): The contract re-computes the winner from declared moves and asserts it matches the TEE-signed winner field. Even a compromised enclave cannot declare an inconsistent result.

  • seal_approve gate — Seal key servers run a dry-run of this function before releasing decryption keys. Decryption is blocked until both players have committed, preventing early reveal.
  • IntentMessage<T> pattern — All TEE payloads are wrapped in a typed intent envelope with scope discriminant + timestamp, preventing cross-context signature reuse.
  • Timeout bounds — Minimum 60s (prevents griefing), maximum 24h (prevents permanent fund locking). Cancel splits stakes 50/50 if both joined.
  • Self-join blocked — Contract rejects Player1 joining their own game.

On-Chain Game Design Lessons

  • Shared objects work well for 2-player games — both players and the resolver can interact with the same Game object
  • Seal IBE with game object ID as identity creates natural per-game encryption scoping
  • The BCS encoding between Rust enclave and Move contract must match exactly — field order in structs is the serialization order
  • Ephemeral keypairs in the enclave mean key rotation is just a redeploy + register_enclave call
