SoLiquid

Bottle recycling rewards platform on Solana: scan QR codes on bottles, earn $SLIQ tokens, redeem partner coupons

Rust / Anchor Solana TypeScript Bun Hono React Native Expo Drizzle ORM SQLite SPL Token

The Challenge

Bottle recycling lacks tangible, immediate incentives.

Existing recycling programs offer vague environmental goodwill but no real-time reward loop. Deposit systems require physical return stations. There is no way to instantly reward someone for recycling a bottle, track their impact, or let local businesses participate as reward partners.

Building an incentive layer on top of physical recycling requires solving several hard problems: preventing double-scan fraud (one bottle scanned twice), verifying QR authenticity without a centralized server bottleneck, minting fungible rewards that users can trade or spend, and creating a marketplace where local businesses offer real coupons. SoLiquid puts the trust layer on Solana and the UX in a mobile app.

Why It Matters

  • Scan-to-earn loop: scan a QR code on a bottle, instantly receive 10 $SLIQ tokens minted on Solana. The entire flow takes seconds.
  • On-chain fraud prevention: each bottle ID creates a unique PDA (Program Derived Address). Attempting to scan the same bottle twice fails at the Solana program level.
  • Cryptographic QR verification: QR codes contain an ed25519-signed payload (bottle_id + nonce). The Solana program verifies the signature on-chain using the instructions sysvar.
  • Partner coupon marketplace: local businesses deposit coupons on-chain. Users burn $SLIQ to redeem. Fully decentralized, no intermediary takes a cut.
  • Full-stack dApp: Anchor smart contract + Bun/Hono backend + React Native/Expo mobile app, all sharing types.

Key Insights

On-Chain Double-Scan Prevention

The PDA seed [b"scan", bottle_id] makes double-scanning impossible at the protocol level. When a user scans a bottle, Anchor's init constraint creates a new ScanRecord account. If that bottle was already scanned, the PDA already exists and the transaction fails. No database check needed, the blockchain enforces uniqueness.

Ed25519 QR Verification

QR codes are signed by the server's ed25519 keypair. The Solana program uses the instructions_sysvar to verify that a preceding ed25519_program instruction validated the signature. This proves the QR was generated by the authorized server without storing a secret on-chain. The verification parses the ed25519 instruction data layout byte-by-byte.

Daily Rate Limiting On-Chain

Each user has a DailyTracker PDA that counts scans per day. The day boundary is computed as unix_timestamp / 86400. When the day rolls over, scans_today resets to 0. The hard cap of 50 scans/day prevents Sybil farming while allowing genuine high-volume recyclers.

Burn-to-Redeem Economics

Coupon redemption burns $SLIQ tokens permanently via token::burn(). This creates deflationary pressure: tokens are continuously minted through recycling but destroyed when users claim rewards. Partners set their own $SLIQ price per coupon, creating a market-driven reward economy.

Checked Arithmetic Everywhere

All arithmetic in the Solana program uses checked_add, checked_mul, and explicit Overflow error codes. No unchecked operations exist in the codebase. This prevents the class of integer overflow vulnerabilities common in smart contracts.
