Claude Swarm

Multi-agent Claude Code orchestration via tmux and shared filesystem

Bash tmux Claude Code Git Worktrees SSH rsync

The Challenge

A single Claude Code session cannot parallelize work.

Complex tasks like "build a REST API with auth, database, and tests" have independent sub-problems that could be worked on simultaneously. But Claude Code runs sequentially in a single terminal, leaving multi-core machines and remote servers idle.

The goal is to launch N Claude Code agents that work on a shared codebase simultaneously, each in its own isolated git worktree, coordinating through a simple file-based messaging protocol. No MCP servers, no external dependencies, just native Read/Write/Bash tools that every Claude Code instance already has.

Why It Matters

  • True parallelism: up to 20 worker agents execute sub-tasks concurrently, each in a separate tmux window with its own git branch.
  • Zero infrastructure: no databases, no message queues, no servers. The entire communication protocol runs on the filesystem via JSON and Markdown files.
  • SSH remote execution: agents can run on a remote VPS or Mac Mini via SSH, with bidirectional workspace sync via rsync every N seconds.
  • Self-coordinating: the orchestrator (Agent #0) breaks down the task, assigns work, monitors progress, and collects results. Workers poll for tasks and report back autonomously.

How It Works

Swarm Launch Flow

USER LAUNCHER TMUX SESSION ./launcher.sh "Build API" init-workspace.sh | | v v Parse args Create /tmp/claude-swarm-{id}/ (workers, project, ├── registry/ SSH host) ├── messages/ | ├── tasks/ v └── artifacts/ Create tmux session | | v +---> Window 0: spawn-agent.sh 0 orchestrator +---> Window 1: spawn-agent.sh 1 worker-1 +---> Window 2: spawn-agent.sh 2 worker-2 +---> Window N: spawn-agent.sh N worker-N +---> Monitor: monitor.sh (live dashboard) | v Auto-accept trust dialogs (sleep 3 + Enter)

Write tasks/task-001-assignee-1.json 3. Execute in isolated worktree 4. Write tasks/task-002-assignee-2.json 4. Save output to artifacts/agent-{id}/ | 5. Update task (status: completed) v 6. Send result message to Agent #0 MONITOR LOOP 7. Go back to step 1 Read registry/agent-*.json Read messages/ (to-0, to-all) COMMUNICATION Handle blockers, reassign messages/{ts}-from-{id}-to-{target}.md When all done: write result.md registry/agent-{id}.json (status updates) Broadcast SWARM_COMPLETE tasks/task-{NNN}-assignee-{id}.json

SSH Remote Mode

LAPTOP REMOTE VPS launcher.sh --ssh user@vps spawn-agent-ssh.sh | | v v Local workspace Remote workspace (rsync) /tmp/claude-swarm-*/ /tmp/claude-swarm-*/ | | +--- sync-workspace.sh ---+ claude --dangerously-skip-permissions | (bidirectional rsync (agents run natively on VPS) | every 5 seconds) v Monitor window (local)

Key Insights

Filesystem as Message Bus

No external infrastructure needed. The entire coordination protocol runs on JSON files in /tmp/. Agents read and write task assignments, status updates, and inter-agent messages using the same Read/Write/Bash tools they already have. This makes the system trivially portable to any machine with tmux and Claude Code.

Git Worktree Isolation

Each agent gets a dedicated git worktree with a unique branch name (swarm-agent-{id}-{timestamp}-{random}). This means agents can modify files in the same repository without merge conflicts during execution. The orchestrator collects results from artifacts/ when all tasks complete.

Template-Based Agent Identity

Each agent receives a dynamically generated CLAUDE.md from agent-claude.md with conditional sections. The orchestrator gets planning and monitoring instructions. Workers get a task-polling loop. Template placeholders ({{AGENT_ID}}, {{WORKSPACE}}) are replaced at spawn time via sed.

Graceful Cleanup

The cleanup script handles four artifact types: /tmp/ workspaces, agent directories, tmux sessions, and git worktree branches. It supports --dry-run, --keep N (preserve recent runs), and --force. Safety check: refuses to delete anything outside /tmp/.

Monitor Dashboard

A dedicated tmux window refreshes every 3 seconds, showing agent statuses with icons ([WORK], [IDLE], [BLKD], [DONE]), the 10 most recent inter-agent messages, task progress, and the final result when the swarm completes. JSON parsing uses python3 for robustness against malformed files during race conditions.
