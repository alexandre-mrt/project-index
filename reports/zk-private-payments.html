The Challenge

On-chain payments are fully transparent. Every transaction reveals the sender, the recipient, and the exact amount. Anyone with a block explorer can trace the full financial history of any address.

Standard ERC-20 transfers and ETH sends create a permanent, public record. Salary payments, business transactions, charitable donations, and personal purchases are all visible to the world. Financial privacy, taken for granted in traditional banking, does not exist on public blockchains by default.

Why It Matters

  • Financial privacy is a right — salary recipients should not have their income visible to every counterparty
  • Business confidentiality — vendors, clients, and competitors can see payment flows between companies
  • Personal safety — large balances linked to a known identity create physical security risks
  • Fungibility — without privacy, coins with certain transaction histories can be discriminated against

How It Works

The system combines two complementary privacy techniques: stealth addresses hide who receives a payment, and confidential amounts hide how much is transferred. Everything is enforced by Groth16 zero-knowledge proofs generated from Circom circuits.

Stealth Address Protocol

  Receiver (one-time setup)              Sender (per payment)
  -------------------------              --------------------

  spending_key (s) ----+                 1. Fetch receiver's viewing pubkey V
  spending_pub  (S=sG) |
                       |                 2. Generate ephemeral secret (r)
  viewing_key   (v) ---+                    ephemeral_pub R = r * G
  viewing_pub   (V=vG) |
                       |                 3. Shared secret via ECDH:
  Register V on-chain -+                    S = r * V  (= v * R for receiver)

                                         4. Stealth pubkey:
                                            P = Poseidon(S.x, S.y) * G + spending_pub

                                         5. Deposit with commitment using P as owner
                                            Announce (R, P, encrypted note data)

  Receiver scans announcements:
    For each (R, P):
      shared = v * R
      derived = Poseidon(shared.x, shared.y) * G + S
      if derived == P  -->  note belongs to me
      Decrypt amount and blinding from encrypted data

Confidential Notes (UTXO Model)

  commitment  = Poseidon(amount, blinding, ownerPubKeyX)
  nullifier   = Poseidon(commitment, spending_key)

  Transfer Circuit (Groth16)
  +---------------------------------------------------------+
  | PUBLIC:  root, nullifier, outCommitment1, outCommitment2|
  |                                                         |
  | PRIVATE: amountIn, blindingIn, ownerPubKeyX,            |
  |          spendingKey, pathElements[], pathIndices[],     |
  |          amountOut1, blindingOut1, ownerOut1,            |
  |          amountOut2, blindingOut2, ownerOut2             |
  |                                                         |
  | CONSTRAINTS:                                            |
  |   1. commitment_in is in Merkle tree (root)             |
  |   2. nullifier = Poseidon(commitment_in, spending_key)  |
  |   3. amountIn == amountOut1 + amountOut2                |
  |   4. All amounts in [0, 2^96) via binary decomposition  |
  |   5. Output commitments correctly computed              |
  +---------------------------------------------------------+

Contract Features

  • Multi-denomination support — owner-managed denomination set for uniform privacy sets
  • Batch deposits — up to 10 notes per transaction
  • Relayer fee system — gasless withdrawals with relayer and fee bound to ZK proof (no front-running)
  • Compliance allowlist — optional depositor restrictions
  • Minimum deposit age — flash loan protection
  • Maximum withdrawal amount — rate limiting
  • Emergency drain — owner-only when paused
  • Chain ID replay protection
  • OpenZeppelin guards — ReentrancyGuard, Pausable, Ownable
  • Deposit receipts — soulbound ERC-721 NFTs as proof of deposit

Key Insights

The core design combines stealth addresses (receiver anonymity) with confidential amounts (hidden values) in a single UTXO system. Neither technique alone provides full payment privacy. Together, they hide both the recipient identity and the transfer amount while maintaining verifiable balance preservation through ZK proofs.

  • Stealth addresses alone hide the recipient but leave amounts visible — an observer can correlate payments by matching deposit and withdrawal values
  • Confidential amounts alone hide how much, but the link between sender and recipient remains visible on-chain
  • Combined — the on-chain record shows only commitments entering a Merkle tree and nullifiers being consumed. No addresses, no amounts
  • 96-bit range proofs cover up to ~79 billion ETH per note (10,000x current supply), preventing overflow attacks while keeping circuit constraints practical
  • Poseidon everywhere — using Poseidon for commitments, nullifiers, Merkle tree hashing, and stealth key derivation keeps constraint counts low inside SNARK circuits
  • Encrypted announcements — note data (amount, blinding) is XOR-encrypted with a Poseidon-derived key from the ECDH shared secret, so only the intended receiver can decrypt
