The Challenge

CI/CD platforms like GitHub Actions, CircleCI, and GitLab CI all live in the cloud. Every push triggers a remote build with cold-start latency, opaque runner environments, and zero offline capability. For solo developers or small teams iterating fast on local code, the feedback loop is unnecessarily slow.

miniCI brings CI/CD to your machine: define pipelines in YAML, execute them in Docker containers locally, and watch logs stream in real-time on a web dashboard. No cloud account, no network dependency, sub-second trigger-to-execution.

Why It Matters

  • Instant feedback — no waiting for remote runners to spin up; pipelines start immediately on your Docker daemon
  • Full isolation — every step runs in its own container with a defined image, environment, and timeout
  • Offline-first — works without internet once images are cached locally
  • Familiar model — YAML-based pipeline definition mirrors GitHub Actions / GitLab CI syntax
  • Developer experience — real-time WebSocket log streaming, polished React dashboard, one-click manual triggers

Pipeline Flow

  1. Define — write a YAML config with stages, steps, images, commands, env vars, timeouts, retries
  2. Register — POST the pipeline via REST API (or trigger via GitHub webhook)
  3. Queue — a run is created and enqueued in the bounded mpsc job queue
  4. Schedule — the scheduler picks it up (respecting max_concurrent via semaphore)
  5. Execute — stages run in dependency-wave order; within a stage, steps run in parallel via JoinSet
  6. Stream — container stdout/stderr is attached before start, broadcast to all WebSocket subscribers, and persisted to SQLite
  7. Report — dashboard updates live with status badges, exit codes, and full log history

Architecture Details

YAML Pipeline Config

name: my-pipeline
stages:
  - name: build
    steps:
      - name: compile
        image: rust:1.85
        commands:
          - cargo build --release
        env:
          RUST_LOG: info
        timeout_secs: 120
        retries: 3
  - name: test
    depends_on: [build]
    steps:
      - name: unit-tests
        image: rust:1.85
        commands:
          - cargo test

Stage Dependency Resolution

Stages declare depends_on edges. The scheduler partitions remaining stages into waves: all stages whose dependencies are satisfied form a wave and execute concurrently. Deadlocks (cyclic or unresolvable graphs) are detected and fail the run immediately.

Key Insights

Real-time log streaming — Container output is attached before the container starts (no missed lines). Logs flow through a Tokio broadcast channel to all WebSocket subscribers simultaneously, while being persisted to SQLite in the same task. The frontend subscribes per-run or globally.

  • Container lifecycle — pull (skip if cached) → create → attach → start → stream logs → wait → force-remove. Cleanup always runs, even on timeout or error.
  • Concurrency control — bounded mpsc channel (100 jobs) prevents memory blow-up; Tokio semaphore caps concurrent runs at max_concurrent
  • Retry with exponential backoff — steps can declare retries: N; backoff starts at 1s and doubles each attempt (capped at 64s)
  • Graceful shutdown — Ctrl+C triggers a signal handler; Axum drains in-flight requests before stopping
  • Trigger modes — manual (dashboard button), webhook (GitHub push events), or scheduled
