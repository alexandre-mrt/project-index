ZKP Credential Verifier

Anonymous credential verification using Nova folding schemes, Sui Move smart contracts, and a full-stack TypeScript/React interface. Prove you hold a credential without revealing who you are.

Nova IVC Rust WASM Sui Move React Hono / Bun

The Challenge

Traditional credential verification exposes the holder's identity. Show your diploma to prove your degree? The verifier now knows your name, birthdate, and student ID. That is a privacy leak with no technical justification.

The goal: let a holder prove they possess a valid credential (issued by a trusted authority) while revealing only the attributes they choose. The verifier learns nothing else.

Selective disclosure — "I have a university degree issued after 2020" without revealing your name, GPA, or institution. The proof is cryptographically bound to the real credential, unforgeable, and reveals zero extra information.

Why It Matters

  • Privacy-preserving identity — KYC, age verification, membership checks without full data exposure
  • Selective disclosure — the holder controls exactly which attributes are visible
  • On-chain verification receipts — immutable proof that verification happened, without leaking credential data
  • No trusted verifier — anyone can verify the math, the verifier does not need to contact the issuer

Step by step

1
Credential issuance — An issuer registers on-chain (Move IssuerRegistry), defines a schema, and issues a credential to a holder. Attribute values stay off-chain; only a CredentialRecord reference goes on Sui.
2
Selective disclosure — The holder selects which attributes to reveal (by index). Hidden attributes are committed using a Pedersen-like field commitment: commit(v, r) = v*G + r*H with domain-separated constants.
3
Nova IVC proof — Each attribute becomes one IVC step in a CredentialCircuit. The circuit uses a binary selector to mux between revealed (accumulate raw value) and hidden (accumulate commitment) paths, keeping a fixed constraint structure across all steps.
4
Compression — The recursive SNARK is compressed into a CompressedSNARK (Spartan + IPA) for efficient verification. Serialized via bincode for transport.
5
On-chain receipt — Off-chain verification result is committed on Sui via the VerificationRegistry. The Move contract stores a VerificationResult receipt and emits a ProofVerified event. Verifiers earn SUI rewards per verification.

Key Insights

Nova IVC for selective disclosure — Each credential attribute is one IVC step. This lets the circuit handle credentials of any size without changing the constraint system. The binary-selector mux pattern ensures the R1CS structure is identical for revealed and hidden attributes, which Nova requires.

No trusted setup — Pallas/Vesta curves with IPA-based polynomial commitments eliminate the need for a ceremony or ptau files. The entire system is transparent: anyone can verify from scratch.

Hybrid on-chain/off-chain model — Full Nova SNARK verification is too expensive for Sui execution. Instead, authorized off-chain verifiers perform the cryptographic check and commit a signed receipt on-chain using a capability pattern (VerifierCap). The Move contracts handle verifier registration, proof status transitions, and a reward pool that pays verifiers per verification.

Browser-native proving — The WASM bridge compiles the entire Rust prover to WebAssembly, letting credential holders generate proofs directly in the browser without sending their private data to any server.
