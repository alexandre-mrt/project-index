GitHub .github

Centralized GitHub org config with reusable CI workflows for TypeScript, Rust, and Sui Move

GitHub Actions YAML TypeScript/Bun CI Rust CI Sui Move CI Dependabot

The Challenge

Every new repository needs CI workflows, and copy-pasting YAML across 20+ repos leads to drift and maintenance hell.

A solo developer working across TypeScript, Rust, and Sui Move repos needs consistent lint, typecheck, build, and test pipelines. When a workflow improvement is found (pinning an action to a new SHA, adding a cache step), it must be applied to every repo individually. Dependabot configuration is the same story.

GitHub's special .github repository enables organization-wide reusable workflows. Any repo in the org can call these workflows with a single uses: line, inheriting best practices automatically. The repo also hosts the GitHub profile README, branch protection recommendations, and a shared Dependabot template.

Why It Matters

  • Write once, use everywhere: 3 reusable workflows cover the entire stack. A new TypeScript repo gets full CI (lint, typecheck, build, test) with a 3-line YAML file.
  • Security-hardened: all third-party actions pinned to commit SHA (not tags), least-privilege permissions: contents: read on every job, concurrency with cancel-in-progress: true.
  • Configurable via inputs: each workflow exposes boolean flags and string overrides (bun-version, rust-version, sui-version, working-directory, custom commands) so repos can customize without forking.
  • Dependabot integration: a template config keeps npm/cargo dependencies and GitHub Actions themselves up to date via weekly PRs with conventional commit prefixes.

Key Insights

SHA-Pinned Actions

Every third-party action is pinned to a full commit SHA, not a tag. Tags can be moved or deleted by the action author. SHA pinning (actions/checkout@34e114876b...) ensures the exact code that was audited is the code that runs. Dependabot automatically proposes SHA updates when new versions are released.

Conditional Job Execution

Each job in the reusable workflows is gated by a boolean input (has-tests, has-lint, has-clippy, etc.) with sensible defaults. A repo that has no tests can set has-tests: false without forking the workflow. The jobs themselves use if: ${{ inputs.has-tests }} for clean conditional skipping.

Concurrency Control

All workflows use concurrency: group + cancel-in-progress: true. When a new commit is pushed to a PR, any running CI for that same PR is automatically cancelled. This saves runner minutes on force-pushed branches and prevents stale results from appearing alongside current ones.

Frozen Lockfile Enforcement

TypeScript workflows use bun install --frozen-lockfile, which fails the build if bun.lockb is out of sync with package.json. This catches accidental dependency changes that were not committed and ensures CI builds are deterministic.
