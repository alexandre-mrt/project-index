ERC-8183 on Sui

Agentic Commerce Protocol — trustless job escrow for AI agent economies

Sui Move Escrow AI Agents ERC-8183

The Challenge

AI agents cannot transact trustlessly. When Agent A wants to buy a service from Agent B, there is no on-chain primitive for escrowed payment, deliverable verification, or dispute resolution. Agents either trust blindly or don't transact at all.

ERC-8183 was published in February 2026 by the Virtuals Protocol team and the Ethereum Foundation dAI team. It defines a Job primitive: a structured, escrowed agreement between a client agent, a provider agent, and a neutral evaluator. This implementation brings the standard to Sui using Move.

Why It Matters

  • Agentic economy — autonomous software agents need a commerce layer to buy and sell services (data, compute, inference) without human intervention
  • Machine-to-machine payments — escrowed SUI locked until deliverable is verified, eliminating counterparty risk between agents
  • Dispute resolution — a neutral evaluator role can approve or reject deliverables, with automatic fund routing in both cases
  • Composable with ERC-8004 — designed for PTB-level composition with the identity/reputation layer (no Move dependency, same-transaction interop)

How It Works

Job Lifecycle (6 States)


  CLIENT creates job            PROVIDER submits work       EVALUATOR decides
       |                              |                          |
       v                              v                          v
  +--------+    fund()    +--------+   submit()   +-----------+
  |  OPEN  | ----------> | FUNDED | -----------> | SUBMITTED |
  +--------+             +--------+              +-----------+
       |                      |                    /         \
       | reject()             | expire()    complete()    reject()
       v                      v              /               \
  +----------+         +---------+    +-----------+    +----------+
  | REJECTED |         | EXPIRED |    | COMPLETED |    | REJECTED |
  +----------+         +---------+    +-----------+    +----------+
   (no funds)        (refund client)  (pay provider)  (refund client)

Escrow Flow


  Client                    Job (shared object)              Provider
    |                             |                             |
    |--- fund(Coin<SUI>) ------> |  escrow: Balance<SUI>       |
    |                             |                             |
    |                             | <--- submit(deliverable) ---|
    |                             |                             |
    |       Evaluator             |                             |
    |          |                  |                             |
    |          |-- complete() --> |--- platform_fee --> Treasury
    |          |                  |--- evaluator_fee -> Evaluator
    |          |                  |--- remainder -----> Provider
    |          |                  |                             |
    |          |-- reject() ---> |--- full refund ---> Client  |

Three Roles

Client creates the job, sets the budget, funds escrow. Provider accepts and delivers work. Evaluator acts as neutral judge: approves (releasing funds to provider) or rejects (refunding client).

Key Insights

Job as on-chain primitive. Each Job is a Sui shared object, mutated by three different parties across its lifecycle. The ACPRegistry singleton tracks global state (job counter, platform fees, admin).

Escrow pattern. Funds are locked as Balance<SUI> inside the Job object itself. No separate vault contract needed. On completion, fees are split in one atomic transaction: platform fee to treasury, evaluator fee to evaluator, remainder to provider.

Evaluator as dispute resolver. The evaluator is set at job creation and cannot be changed. They alone can approve or reject funded/submitted jobs. This creates a simple but effective arbitration layer for agent commerce without complex governance.

Safety mechanisms. Expiry-based refunds are permissionless (anyone can trigger after timeout). Move's ownership model eliminates reentrancy. Fee math uses basis points with overflow protection (platform + evaluator must be under 100%).
