The Challenge

HTTP status code 402 Payment Required was reserved in 1999 for "future use." Over two decades later, the web still has no native payment layer. Every resource is either free, gated behind a subscription, or monetized through ads and data harvesting.

The x402 protocol finally implements HTTP 402. A server responds with 402 Payment Required plus machine-readable payment requirements. The client signs a transaction, attaches it to the retry request, and the server settles on-chain before serving the resource.

Coinbase published the reference implementation for EVM and SVM. This package, @x402/sui, extends the protocol to the Sui blockchain.

Why It Matters

  • Micropayments — charge $0.001 per API call without credit card minimums or payment processor fees eating the margin
  • Pay-per-use APIs — LLM inference, data feeds, premium endpoints can accept payment inline with the HTTP request
  • No accounts, no subscriptions — a wallet and a signed transaction is all you need
  • Machine-to-machine payments — AI agents can autonomously pay for resources via the MCP integration
  • Stablecoin settlement — payments denominated in USDC, no volatility risk for either party

No custom Move contracts. The payment uses native Sui Programmable Transaction Blocks (PTBs): SplitCoins to extract the exact USDC amount, TransferObjects to send it to the resource owner. The facilitator dry-runs the PTB to verify correctness before executing.

Architecture

MCP Integration

The createSuiPaymentWrapper higher-order function wraps any MCP tool handler with x402 payment logic. When an AI agent calls a paid tool without payment, it receives a structured 402 error with PaymentRequirements. The agent signs a transaction and retries with payment metadata in _meta["x402/payment"].

Security Model

Verification pipeline (facilitator):

  • Cryptographic signature verification via verifyTransactionSignature
  • Dry-run simulation to validate transaction effects before execution
  • Balance change analysis: exact amount, correct recipient, correct asset
  • Signer/payer identity match to prevent signature substitution
  • Side-effect detection: rejects PTBs with unexpected balance changes (prevents injection attacks)
  • Duplicate settlement cache with TTL-based LRU eviction (prevents replay)
  • Request body size limit (1MB) on the facilitator HTTP server

Key Features

  • Sponsored transactions — facilitator can cover gas fees via a gas station URL, enabling gasless payments for end users
  • Multi-token support — USDC by default, but the token registry and coin metadata cache support any Sui fungible token
  • Money parsers — pluggable price parsing: treat numbers as USD and auto-convert to USDC base units, or use custom parsers
  • Standalone facilitator server — createFacilitatorServer() spins up a Bun HTTP server with /supported, /verify, and /settle endpoints
  • Registration helpers — one-liner integration with x402Client, x402Facilitator, and x402ResourceServer

Key Insights

Why Sui for x402?

  • Sub-second finality — payments confirm in ~400ms, fast enough to feel synchronous within an HTTP request/response cycle
  • Low transaction fees — fractions of a cent per transaction make micropayments economically viable (unlike EVM L1 where gas can exceed the payment)
  • Programmable Transaction Blocks — split coins + transfer in a single atomic transaction, no custom smart contract needed. Dramatically simplifies deployment and auditing.
  • Native USDC — Circle-issued USDC on Sui means stablecoin settlement without bridging risk
  • Object model — Sui's object-centric architecture means coin objects can be split and transferred directly, with parallel execution for non-overlapping objects

The MCP angle. With AI agents increasingly calling tools autonomously, x402 on Sui provides the payment rail. An agent that needs premium data can pay for it inline with the MCP tool call, no human approval loop needed. The structured 402 error tells the agent exactly what to pay, and the agent's wallet handles the rest.
