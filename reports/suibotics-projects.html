The Challenge

Physical robots today operate in isolated silos. Renting a robot requires centralized middlemen. Coordinating multiple users controlling the same robot has no trustless mechanism. And there is no on-chain incentive layer to prioritize, queue, or pay for robot actions.

The core problem is trustless robot economies. How do you rent a robot without trusting a platform? How do you fairly share control among multiple users? How do you create economic incentives around robot actions without a central authority?

Suibotics solves this with three interlocking Sui Move projects. A rental marketplace with escrow and receipt NFTs. A multiplayer controller with an on-chain fairness queue. And a token-gated action queue that creates economic incentives for robot task prioritization.

Why It Matters

DePIN Meets Robotics

Decentralized Physical Infrastructure Networks (DePIN) have exploded for sensors, wireless, and compute. Robotics is the next frontier: machines that physically interact with the world, generating real economic value. Suibotics provides the on-chain primitives to make robot economies permissionless.

  • Robot rental as a service — Owners list robots with hourly rates, renters pay in TREAT tokens with automatic escrow and pro-rata refunds. No platform fee, no middleman.
  • Fair multiplayer access — Multiple users queue for robot control with on-chain turn enforcement. Timeouts prevent hogging. Every control transfer is an auditable on-chain event.
  • Tokenized action queues — Users pay COOKIE tokens to enqueue robot actions. The queue creates a market: higher demand means more fees for operators. Actions are dequeued and executed with full on-chain history.
  • Composability — All three projects share the same object model. A rented robot could use the fairness queue for multi-user sessions, with actions paid via the token queue.

Smart Contracts

Robot Rental Platform (4 modules)

  • treat_token — Custom fungible token with 9 decimals. Shared FaucetState with rate-limited minting (5 claims per epoch, 100 TREAT each). Treasury cap locked in faucet.
  • robot_registry — Shared registry mapping robot IDs to RobotInfo (name, capabilities, hourly rate, owner, availability, earnings). OwnerCap pattern for access control. Package-visible helpers for escrow integration.
  • rental_escrow — The core marketplace. Creates rentals with TREAT payment held in escrow. Pro-rata cost calculation (ceiling to nearest hour). Automatic refund of unused time. Mints RentalReceipt NFTs on completion. Admin force-end with full refund.
  • command_auth — Ed25519 challenge-response authentication. Generates 32-byte challenges (8-byte nonce + 24-byte address prefix). On-chain signature verification gates robot command execution. One-time challenge consumption prevents replay.

Multiplayer Robot Controller (2 modules)

  • fairness_queue — On-chain FIFO queue (max 100 players). Turn-based control with configurable duration (default 60s). Voluntary release or admin force-release on timeout. Full event emission for every state transition.
  • robot_state — Shared object tracking position (x,y on 100x100 grid), direction (N/E/S/W), and status (idle/walking/sitting/standing/waving). 6 command types with grid-wrapping movement. Only the current controller (verified via fairness queue) can send commands.

Action Queue Tokenomics (2 modules)

  • cookie_token — COOKIE token with 9 decimals. Faucet with 3 claims per epoch (10 COOKIE each). Time-based epoch calculation (86,400,000ms per epoch). Shared treasury cap.
  • action_queue — Shared queue where users pay 1 COOKIE per action. 5 action types with up to 256 bytes of parameters. Admin dequeues and executes. 1000-entry history with automatic rotation. Fee withdrawal for queue operator.

Key Insights

Escrow with pro-rata refunds is the right rental primitive. The rental escrow locks the maximum cost upfront and refunds unused time on early return, calculated to the nearest hour ceiling. This protects both parties: the owner is guaranteed payment, and the renter never overpays. Receipt NFTs provide an immutable rental history.

Ed25519 challenge-response on-chain is elegant for robot command auth. Instead of relying on wallet signatures alone, the command_auth module issues 32-byte challenges and verifies Ed25519 signatures on-chain. Each challenge is consumed after use, preventing replay attacks. This pattern works for any IoT device that needs cryptographic command authorization.

On-chain fairness queues solve the shared-robot problem. When multiple users want to control the same robot, the fairness queue enforces FIFO ordering with time-limited turns. The 60-second default with admin force-release handles both voluntary handoffs and AFK users. Every control transfer is an on-chain event, making the entire history auditable.

Token-gated action queues create natural market dynamics. Paying 1 COOKIE per action creates backpressure: when the queue is long, users think twice about low-priority actions. The admin dequeue pattern separates payment (on-chain) from execution (off-chain), which is the right split for physical robot actions that cannot be verified on-chain.

The three projects are composable by design. A robot registered in the rental platform could use the fairness queue for multi-user sessions during a rental period, with individual actions paid via the COOKIE token queue. The shared Sui object model and capability pattern make cross-project integration straightforward.
