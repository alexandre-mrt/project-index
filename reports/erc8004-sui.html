ERC-8004 on Sui

Trustless agent identity, reputation, and validation on Sui blockchain

Sui Move On-Chain Identity Reputation Validator Attestation ERC-8004

The Challenge

AI agents operate without verifiable identity.

There is no standard way to verify who deployed an agent, assess its track record, or confirm that a third party has audited its behavior. Trust is implicit, not provable.

As autonomous AI agents proliferate across DeFi, data markets, and service orchestration, a fundamental gap emerges: how do you trust a piece of software that acts on its own? Today, agents self-report their capabilities. There is no on-chain record of who created them, how they have performed, or whether independent validators have attested to their reliability.

Without a trust layer, every interaction with an agent is a leap of faith. Malicious agents can impersonate legitimate ones. Good agents have no way to build a verifiable reputation. The agent economy cannot scale without a shared identity and trust primitive.

Why It Matters

  • Trustless discovery -- agents register on-chain with a unique ID, URI, and metadata. Any protocol or user can verify an agent's identity without relying on centralized directories.
  • Reputation as a primitive -- feedback from clients is stored on-chain with signed scores, tags, and revocation support. An agent's track record becomes a composable on-chain asset.
  • Validator attestation -- designated validators can formally attest to agent behavior with scored responses (0-100). This creates a decentralized quality assurance layer.
  • Cross-organizational trust -- agents from different teams, DAOs, or companies can interact with each other using a shared trust framework rather than bilateral agreements.

Key Insights

NFT-Based Agent Identity

Each agent is an owned Sui object rather than a registry entry. This means agent identity is composable: it can be transferred, wrapped by other modules, or used as a proof-of-ownership in cross-protocol interactions. The owner controls the object directly through Sui's native authorization, with no require(msg.sender == owner) checks needed.

Validator Attestation Pattern

Validation follows a request-response model with designated validators. The agent owner requests validation by specifying a validator address and a request hash. Only that exact validator can respond, with a score from 0 to 100. Multiple responses to the same request are allowed, enabling progressive validation states where a validator can update their assessment over time.

Reputation as an On-Chain Primitive

Feedback is not just a number. Each entry carries two tag dimensions (e.g., tag1: "accuracy", tag2: "defi") enabling structured queries. The get_summary function supports filtering by client addresses and tags, returning aggregated net scores. Self-feedback is blocked at the protocol level: the identity registry is cross-referenced to verify the caller is not the agent owner.

Signed Integers Without Signed Integers

Move has no native signed integer type. The protocol represents feedback values as u64 + is_negative bool, and the summary function accumulates positive and negative totals separately before computing the net result. Decimal precision is preserved via a value_decimals field (0-18), matching Ethereum's token decimal convention.
