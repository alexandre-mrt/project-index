Jarvis Phone

Telegram bot for monitoring and controlling Claude Night Shift and Long Run sessions from your phone

TypeScript Bun Telegram Bot API Groq Whisper Claude Code CLI Biome macOS TTS

The Challenge

Night shift sessions run for hours unsupervised, with no way to check progress or intervene from a phone.

Claude Code's night-shift and long-run modes execute autonomously on a laptop or Mac Mini. When something goes wrong (a task fails, a blocker appears, a question needs answering), the only way to know is to sit at the terminal and watch. That defeats the purpose of autonomous overnight execution.

Jarvis Phone bridges the gap: a Telegram bot that watches project state files, sends real-time notifications when events occur, and lets the user respond with text or voice messages from their phone. It can also spawn Claude Code processes remotely, including launching night shift sessions on any monitored project.

Why It Matters

  • Real-time monitoring: a filesystem watcher detects state file changes across all projects and dispatches Telegram notifications for task completions, failures, new problems, session endings, and phase transitions.
  • Voice-first interaction: send a voice message from your phone, Groq Whisper transcribes it, and the instruction is written to VOICE_INSTRUCTIONS.md for the running Claude session to pick up.
  • Remote process management: spawn Claude Code processes from Telegram, launch night-shift sessions on specific projects, list running processes, and kill them by PID.
  • Quiet hours: notifications are suppressed between 23:00 and 09:00 and queued until morning, so the phone stays silent at night while sessions keep running.

How It Works

Event Detection Pipeline

FILESYSTEM WATCHER TELEGRAM ~/projects/ project-a/ NIGHT_SHIFT_STATE.md ----+ LONG_RUN_STATE.md -------+---> node:fs.watch NIGHT_SHIFT_PROBLEMS.md -+ (recursive) project-b/ | NIGHT_SHIFT_STATE.md ----+ v ... | Diff Detection | Compare prev vs current content | Count [x] tasks, [NEEDS-REWORK] | Match phase pattern, detect finish | v Emit WatcherEvent { type, projectName, detail } | v Telegram Bot API sendMessage(chatId, formatted HTML)

Voice Interaction Flow

USER (Phone) BOT CLAUDE SESSION Record voice message | v Telegram sends voice ------> getFile(file_id) Download .ogg audio | v Groq Whisper API Transcribe to text | v Write to project: VOICE_INSTRUCTIONS.md ------> Claude picks up instruction --- Reply to question --- Voice reply to question ----> Transcribe via Groq | v parser/writer.ts Write answer to LONG_RUN_QUESTIONS.md (append under question) ------> Claude reads answer

Process Management

TELEGRAM COMMAND PROCESS MANAGER "/projects" ----------------> List active sessions Return inline keyboard [Project A] [Project B] "night shift on myapp" ------> isNightShiftRequest() Detect pattern + project name | v spawnClaude() child_process.spawn("claude", ["--dangerously-skip-permissions", "-p", "/night-shift"], { cwd: projectPath }) | +---> stdout/stderr -> onOutput +---> on close -> onExit +---> Regular: 5min timeout Night shift: no timeout

Key Insights

Diff-Based Event Detection

The watcher compares previous and current file content to detect specific events. It counts [x] checkboxes to detect task completions, matches [NEEDS-REWORK]/[BLOCKED] to detect failures, parses ### Current Phase for iteration changes, and checks for the transition from (in progress) to Finished: for session completion. This is more reliable than filesystem event types alone.

Night Shift vs Regular Process Handling

The process manager distinguishes between night-shift spawns (no timeout, long-running) and regular instructions (5-minute timeout, killed on expiry). Night-shift detection uses regex patterns on the incoming text, extracting both the intent and the target project name from messages like "night shift on myapp".

Message Auto-Cleanup

The MessageTracker in middleware.ts keeps a rolling window of sent message IDs persisted to disk. When the count exceeds a threshold, the oldest messages are batch-deleted via Telegram's deleteMessages API (up to 100 per call, with fallback to individual deletes). This keeps the chat clean during long sessions that generate many notifications.

Immutable Types Throughout

All interfaces use readonly properties. The watcher emits immutable WatcherEvent objects. The bot returns immutable TelegramUpdate arrays. The process manager's listProcesses() returns a readonly array. This prevents accidental mutation of shared state between the watcher, bot, and process manager.
